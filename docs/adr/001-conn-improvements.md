# ADR-001: Улучшения системы подключений и управления пользователями

**Дата:** 2026-04-19  
**Статус:** Применено

---

## Контекст

В ходе работы над ботом были выявлены несколько несоответствий между тем, как бот работает, и тем, чего ожидает администратор. Проблемы касались:

- Отсутствия различий между конфигурациями одного пользователя в панели 3x-ui
- Того, что оплата не подтверждалась автоматически в момент выдачи платного конфига
- Наличия функционала «Даты оплат», который не нёс практической ценности
- Отсутствия возможности добавить пользователя вручную, не дожидаясь пока тот сам напишет боту
- Того, что администраторы попадали в систему напоминаний об оплате наравне с обычными клиентами

---

## Что было сделано и почему

### 1. Нумерация конфигов в панели 3x-ui

**Было:** все конфигурации назывались «VPN» — при нескольких конфигах одного пользователя в панели 3x-ui было невозможно понять, кому какой принадлежит.

**Стало:** при создании конфига бот автоматически считает, сколько конфигураций у пользователя уже есть, и присваивает следующий порядковый номер: 1, 2, 3 и так далее. Имя задаётся без участия администратора.

**Технически:** перед созданием клиента в 3x-ui делается запрос на получение списка уже существующих клиентов этого пользователя в inbound. Новый клиент получает имя `count + 1`. Параметр `label` убран из сигнатуры метода `Create`, так как стал излишним.

---

### 2. Автоматическое подтверждение оплаты при выдаче платного конфига

**Было:** когда администратор подтверждал оплату и выдавал конфиг, в системе ежемесячных платежей запись за текущий месяц не появлялась. Формально пользователь получал подключение, но система считала его не оплатившим.

**Стало:** сразу после выдачи платного конфига бот автоматически помечает текущий месяц как оплаченный. Пользователь не получит напоминание об оплате за тот же месяц, в котором подключился.

Дополнительно исправлена логика флага `isFree`: для платных подключений он теперь корректно выставляется в `false`, что означает — подключение будет отслеживаться в системе ежемесячных платежей.

---

### 3. Удаление функционала «Даты оплат»

**Было:** в панели администратора существовала кнопка «📅 Даты оплат», позволявшая вручную задать дату оплаты для конкретного подключения. На основе этой даты планировщик раз в месяц отправлял напоминание.

**Почему убрали:** эта система дублировала уже существующий механизм ежемесячных платежей (таблица `payments`), при этом не вписывалась в него — создавала отдельный, параллельный цикл напоминаний. Поддерживать оба потока было избыточно.

**Что осталось нетронутым:** колонка `last_paid_at` в базе данных сохранена — данные, записанные ранее, не теряются. Просто бот больше не читает и не пишет в неё.

**Технически удалено:**
- Кнопка «📅 Даты оплат» из главного меню администратора
- Три экрана: список пользователей, список подключений, ввод даты
- Функция `sendPayDateReminders` из планировщика
- Методы `SetConnLastPaidAt` и `GetConnsWithDuePaidReminder` из всех слоёв (usecase, репозиторий)
- Состояние сессии `StateSetPayDate` и связанные ключи

---

### 4. Ручное добавление пользователей администратором

**Было:** администратор не мог выдать конфиг человеку, который ещё не написал боту. Список клиентов в разделе «Подключения» показывал только тех, кто самостоятельно запустил бота.

**Стало:** в разделе «🔗 Подключения» появилась кнопка «➕ Новый клиент». Администратор вводит числовой Telegram ID клиента — бот пытается получить имя и username через Telegram API. Если получилось — создаёт пользователя с полными данными. Если нет (человек заблокировал ботов или никогда ни с кем не взаимодействовал) — создаёт запись только с ID, имя заполнится автоматически, когда пользователь сам напишет боту.

После создания сразу открывается страница подключений нового клиента, откуда можно выдать конфиг.

**Технически:** добавлены новый callback `adm_conn_new`, состояние сессии `StateAddManualUser`, обработчики `HandleAdminConnNewUser` и `HandleSessionAddManualUser`. Повторное создание безопасно — метод `RegisterOrGet` вернёт существующего пользователя, если тот уже есть.

---

### 5. Администраторы исключены из системы оплаты

**Было:** если администратор заводил себе VPN-конфиг, он попадал в систему напоминаний наравне с обычными клиентами: получал уведомления об оплате, а его подключение могло быть автоматически заблокировано за просрочку.

**Стало:** конфиги администраторов всегда создаются бесплатными, вне зависимости от того, какой тип выбрал сам администратор при создании. Планировщик пропускает администраторов в трёх местах:

- При отправке напоминаний о неоплаченных подключениях
- При отправке ежемесячных напоминаний об оплате
- При автоматической блокировке подключений за просрочку

**Технически:** список `adminIDs` из конфигурации передаётся в `connectionUseCase` и `Scheduler`. Проверка `isAdmin(userID)` выполняется до любых действий, связанных с оплатой. Новый параметр добавлен в конструкторы обоих компонентов.

---

## Что не менялось

- Схема базы данных не изменилась (колонки не добавлялись и не удалялись, все миграции идемпотентны)
- Логика работы с 3x-ui не затронута
- Пользователи-«друзья» (`is_free_friend`) работают по-прежнему
- Все остальные разделы панели администратора (рассылки, оплаты, клиенты) не изменились
